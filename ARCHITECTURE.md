# live-translator 架构原理文档

## 0. 这个项目做什么

实时监听电脑系统声音（比如正在播放的英文视频），把听到的英语转成文字，再翻译成中文。

```
你电脑在播视频 ──→ 抓取系统声音 ──→ 检测到有人说话 ──→ 语音转英文 ──→ 英文翻中文 ──→ 输出
```

全程在你自己电脑上跑，不需要联网。

---

## 1. 声音是怎么变成数据的

### 1.1 声音的本质

声音是一连串的空气振动。振幅大 → 声音响，频率高 → 音调高。

```
振幅 ↑     ●●●
      │   ●●  ●●
      │  ●      ●
      │─●────────●─→ 时间
      │           ●
      │            ●●
```

### 1.2 采样（Sampling）

电脑没法存"连续的波"，只能每隔一个固定的时间间隔记录一次振幅。这就是**采样**。

我们用的是 **16000 Hz**，意思是每秒记录 16000 次。为什么选 16000？因为人类语音最高频率大概 8000Hz，根据奈奎斯特定理（采样率 ≥ 信号最高频率 × 2），16000 足够了。比 CD 音质（44100Hz）低很多，但对语音识别来说够用且省算力。

```
连续波：  ╭─╮  ╭─╮
         ╱  ╲╱  ╲╱
采样后：  ●   ●   ●   ← 每个点是一个数字 (f32, 范围 -1.0 ~ 1.0)
         ●• •● •● •●
```

### 1.3 声道和位深

- **单声道（Mono）**：只有一路声音。立体声（Stereo）是两路，我们不需要，直接取平均合并。
- **f32 格式**：每个采样点存为 32 位浮点数，范围 -1.0（最响的负振幅）到 1.0（最响的正振幅）。静音是 0.0。

### 1.4 音频块（Chunk）

我们不处理单个采样点，而是一次处理 **512 个采样点**（约 32 毫秒的声音）。这 512 个点组成一个"块"（chunk），后续所有处理都以块为单位。

```
一秒的声音 = 16000 个采样点 ≈ 31 个 chunk
一个 chunk  = 512 个采样点 ≈ 32 毫秒
```

## 2. 怎么知道有人在说话 —— VAD

VAD = Voice Activity Detection = 语音活动检测

### 2.1 能量检测

原理极其简单：算一个 chunk 里所有采样点的**均方根（RMS）**，如果大于某个阈值，就是有人在说话。

```
RMS = sqrt( (a₁² + a₂² + ... + a₅₁₂²) / 512 )

如果 RMS > 0.005 → 在说话
如果 RMS < 0.005 → 静音
```

0.005 这个阈值是通过实验调的。太低了会把空调声当说话，太高了轻声说话检测不到。

### 2.2 切分成段

现实中的语音不是一口气说完的。我们通过"说话→停顿→再说话"的模式来切分：

```
██████████░░░░░░██████░░░░░░░░████████████
  说了一段     停顿   又说一段    长停顿
              ↓                    ↓
          切一个 segment       切一个 segment
```

具体规则：
- 检测到说话开始 → 累积音频
- 静音超过 **1 秒** → 认为这段话说完了，切出来
- 连续说话超过 **10 秒** → 强制切一段（防止 memory 爆掉，也防止输入太长 Whisper 处理不了）
- 说话时长不足 **0.15 秒** → 忽略（可能是噪音）

### 2.3 实时预览与最终识别

说话过程中有两个时机触发 ASR：

**Partial Recognition（实时预览）**：说话进行中，每 500-1000 毫秒，把当前累积的音频取最近 5 秒送去 ASR。这给你"实时字幕"的效果——视频一边播，字幕一边更新。但因为是半截话，结果可能不准。

**Final Recognition（最终识别）**：VAD 判断一段话说完了，把这段完整音频（最长 10 秒）送 ASR，得到最准确的结果。这个结果会送去翻译。

## 3. 语音怎么转成文字 —— ASR（Whisper）

这是整个系统最核心、也最慢的部分。

### 3.1 Whisper 是什么

OpenAI 开源的语音识别模型。它的结构是 **Encoder-Decoder Transformer**。

我们用的是 **whisper-small**：
- 参数量：约 461M（4.6 亿个浮点数权重）
- 模型文件大小：923MB
- Encoder：12 层 Transformer
- Decoder：12 层 Transformer
- d_model：768（每层处理的向量维度）

模型越大越准也越慢：

| 型号 | 参数 | 大小 | 效果 |
|------|------|------|------|
| tiny | 39M | 145MB | 几乎不能用 |
| base | 74M | 277MB | 偶尔识别几个词 |
| **small** | **461M** | **923MB** | **可用，速度精度平衡** |
| medium | 769M | 1.5GB | 更准但更慢 |
| large-v3 | 1.5B | 2.9GB | 最准，需要 GPU |

### 3.2 第一步：把声音转成"声音图片"——梅尔频谱

Whisper 不是直接听原始声波，而是先把声音转成一种叫**梅尔频谱（Mel Spectrogram）**的东西。你可以把它理解成一张对"声音"拍的照片。

**为什么叫"声音图片"？**
- 横轴 = 时间
- 纵轴 = 频率（低音在下，高音在上）
- 颜色深浅 = 能量大小（红=响，蓝=轻）

```
频率 ↑
高音 │  ░░    ░░
     │ ░░░░  ░░░░
中音 │████████████░░
     │██████████████
低音 │██████████████
     └──────────────→ 时间
        "Hel-  -lo"
```

**怎么做出来的？**一共六步：

| 步骤 | 干什么 | 简单理解 |
|------|--------|----------|
| 1. 分帧 | 把音频切成每 25ms 的小窗口，窗口之间滑动 10ms | 把一整条声音切成很多小片段 |
| 2. 加窗 | 每个窗口乘以汉宁窗 | 让每小段的开头结尾平滑过渡，不会突然断掉 |
| 3. FFT | 快速傅里叶变换 | 把"时域"（振幅随时间变）转成"频域"（有哪些频率成分） |
| 4. 功率谱 | 取绝对值平方 | 算出每个频率有多响 |
| 5. 梅尔滤波器组 | 80 个三角滤波器 | 模拟人耳：低频敏感（细分辨），高频不敏感（粗分辨） |
| 6. 取对数 | `log10(x)` | 模拟人耳对响度的对数感知（10倍能量 ≈ 感觉响了一倍） |

最终得到一个 **80 × N 的矩阵**（80 个梅尔频带 × N 个时间帧 = "声音图片"）。

### 3.3 第二步：理解"声音图片"——Encoder

把"声音图片"喂给 Whisper 的 Encoder。Encoder 的作用是理解音频内容，提取语义特征。

```
梅尔频谱 (80×1500 的"声音图片")
    │
    ▼
┌─────────────────────┐
│  卷积层 ×2           │  用 3×3 小卷积核扫过图片，压缩时间维度（stride=2）
│  (CNN, 类似图像处理)  │  1500 帧 → 750 帧，提取局部特征
└─────────┬───────────┘
          ▼
┌─────────────────────┐
│  Transformer ×12 层  │  双向自注意力：每一帧同时看所有其他帧
│  (理解整段音频的上下文)│  "当前这个音是 'e'，结合前面 'h' 和后面 'llo'，
│                      │   这整个词应该是 'hello'"
└─────────┬───────────┘
          ▼
    encoder 输出: 750×768 矩阵
    (每个时刻一个 768 维的向量，包含了该时刻完整的声音特征)
```

关键点：Encoder 的注意力是**双向的**——处理第 3 秒时，同时看第 1 秒和第 10 秒。这就是为什么它**不能流式处理**——必须一次性拿到整段音频才能开始工作。这也是为什么"实时"Whisper 本质上是不可能的。

### 3.4 第三步：逐个词拼出来——Decoder

Decoder 的工作是把 Encoder 提取的音频特征"翻译"成文字 token。这个过程是**自回归**的：一次只预测一个字，然后把刚预测的字加回去，再预测下一个字。

```
Encoder 输出 (750×768 音频特征)
         │
         ▼
┌────────────────────────────────────┐
│  初始 prompt tokens:                │
│  <SOT> <en> <transcribe> <notimestamps>  │
│  "开始  英语   转录        不要时间戳"      │
└────────────────┬───────────────────┘
                 ▼
┌────────────────────────────────────┐
│  Decoder ×12 层, 循环 100 次:       │
│                                    │
│  自注意力: "我之前说了哪些词？"        │
│  交叉注意力: "Encoder 听到了什么声音？"   │  ← 这里把声音和文字连起来
│                                    │
│  输出: "下一个词的概率分布"            │
│  "the" → 0.78                      │
│  "a"   → 0.12                      │
│  "dog" → 0.05                      │
│  ...                               │
│  选概率最大的: "the"                 │
│  追加到序列: [SOT, en, ..., "the"]  │
│  回到开头，继续预测下一个词            │
└────────────────┬───────────────────┘
                 ▼
          重复直到输出 EOT (结束符)
                 ▼
         "These chocolate eggs ..."
```

### 3.5 什么是 Token

Whisper 的"词汇表"有 50258 个 token。每个 token 是一个 ID（数字），对应一个或几个字符。

```
Token ID → 字符串（Ġ = 空格前缀）
   12    → "-"
   286   → "ĠI"       解码后: " I"
   366   → "Ġthe"     解码后: " the"
  6215   → "Ġchocolate"
  50359  → "<|transcribe|>"   (特殊控制 token)
  50257  → "<|endoftext|>"     (结束符 EOT)
```

### 3.6 从 token 还原文字

```
Token 序列: [286, 366, 6215, 50257]
    ↓ 查词表
字符串: "ĠI" "Ġthe" "Ġchocolate"
    ↓ Ġ → 空格
文字: "I the chocolate"
    ↓ 去首尾空格
结果: "I the chocolate"
```

## 4. 英文怎么翻成中文 —— NMT

### 4.1 翻译模型

**Marian MT**，专为英译中训练的 Transformer：
- 模型大小：约 300MB
- d_model=512，Encoder 6 层，Decoder 6 层
- 用 SentencePiece 分词（把句子切分成子词）

### 4.2 翻译流程

```
英文: "These chocolate eggs"
    │
    ▼
┌──────────┐
│ 分词     │  SentencePiece: ["_These", "_chocolate", "_eggs"]
└────┬─────┘
     ▼
┌──────────┐
│ Encoder  │  Transformer ×6 层，理解英文句子含义
└────┬─────┘
     ▼
┌──────────┐
│ Decoder  │  Transformer ×6 层，自回归生成中文（和 Whisper Decoder 原理一样）
└────┬─────┘
     ▼
  中文: "这些巧克力蛋"
```

### 4.3 为什么翻译不慢

- 翻译模型的输入是一句话（几十个 token），不是 1500 帧音频
- Encoder 和 Decoder 各只有 6 层（Whisper 有 12 层）
- 实测：**翻译 ~0.5 秒，ASR ~7 秒**。翻译不是瓶颈。

### 4.4 并行架构

ASR 和翻译跑在**两个独立线程**里，用 channel 传递消息：

```
线程 1 (Pipeline):        线程 2 (Translator):
  ...                           翻译 segment_1 ──→ 输出
  ASR segment_2 ──→ channel ──→ 翻译 segment_2 ──→ 输出
  ASR segment_3 ──→ channel ──→ 翻译 segment_3 ──→ 输出
```

翻译 segment_1 的同时，ASR 已经在处理 segment_2 了。两者重叠执行。

## 6. 为什么慢 —— CPU vs GPU

### 6.1 计算量分析

Transformer 的核心操作是"注意力"——让序列中的每个位置和所有其他位置做比较。

```
Encoder 注意力: 750 帧 × 750 帧 = 56 万次 pairwise 比较
               × 12 层 = 675 万次
               × 12 个注意力头 = 8100 万次

Decoder 注意力: ~30 token × 30 token = 900 次
               × 12 层 = 10800 次
```

Encoder 是计算大头。56 万次比较 × 12 层 = 大量的矩阵运算。GPU 天生适合做并行矩阵运算（几千个核心同时算），CPU 只能串行（或者有限的 16 核并行）。

### 6.2 实测数据（10s 音频，AMD 16 核 CPU，whisper-small）

| 阶段 | 耗时 |
|------|------|
| Mel 频谱转换 | < 0.1s |
| Encoder | ~4-6s |
| Decoder（~28 tokens） | ~2-3s |
| **ASR 总计** | **~6-9s** |
| 翻译 | ~0.5s |

ASR 是绝对瓶颈。10 秒的音频需要 6-9 秒处理，远达不到实时。

### 6.3 为什么没有 GPU 加速

`candle` 支持 CUDA（NVIDIA GPU），但我们编译时没用 GPU feature：
- 需要安装 CUDA toolkit、cuDNN
- 模型要转成 GPU 格式
- 这台机器没有 NVIDIA GPU

有了 GPU，同样的 whisper-small 推理可以压到 **1-2 秒**，接近实时。
